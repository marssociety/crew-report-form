## Common Patterns

### Shared header fields

All 11 role-specific report tables share these columns:

| Column | Type | Nullable | Description |
|--------|------|----------|-------------|
| `id` | TEXT (UUID) | NOT NULL | Primary key, generated by app via `uuidv4()` |
| `crew_number` | TEXT | NOT NULL | e.g., "301" |
| `position` | TEXT | nullable | Crew role/position title |
| `prepared_by` | TEXT | nullable | Name of person who prepared the report |
| `report_date` | TEXT | nullable | Date string |
| `sol` | INTEGER | nullable | Sol (mission day) number |
| `email_subject` | TEXT | nullable | Generated email subject line |
| `email_body` | TEXT | nullable | Generated email body text |
| `created_at` | DATETIME | DEFAULT CURRENT_TIMESTAMP | Record creation time |

> **Note**: In schema.sql the `id` column is declared `INTEGER PRIMARY KEY AUTOINCREMENT` for 10 of the 11 tables. However, the app code inserts UUID strings into this column. SQLite accepts this silently due to type affinity. For PostgreSQL, these must be changed to `TEXT PRIMARY KEY`.

### ID generation

- All role-specific reports: UUID v4 generated in JavaScript (`uuidv4()`)
- Legacy crew_reports: `report_id TEXT PRIMARY KEY` (UUID provided by client)

### Child table pattern

Child tables use:
- `SERIAL` / `INTEGER PRIMARY KEY AUTOINCREMENT` for their own `id`
- A foreign key column referencing the parent report's UUID `id`
- `ON DELETE CASCADE` on all foreign keys

| `eva_participants` | crewReportRepository.ts:98 | eva_data_id, participant_name |
| `eva_objectives` | crewReportRepository.ts:107 | eva_data_id, objective |
| `resources` | crewReportRepository.ts:128 | report_id, water_usage_liters, power_usage_kwh, food_consumption |

These code paths will fail at runtime if the legacy form tries to save EVA or resource data. The matching tables from the schema (`resource_usage`, `environmental_data`, EVA waypoint tables) have different column names.

Additionally, `crewReportRepository.ts` inserts a `status` column into `crew_members`, but the schema only defines `name`, `role`, and `affiliation`.

## Notes for PostgreSQL Migration

### Type changes required

| SQLite | PostgreSQL | Affected |
|--------|-----------|----------|
| `INTEGER PRIMARY KEY AUTOINCREMENT` | `TEXT PRIMARY KEY` | 10 parent report tables (id receives UUID) |
| `INTEGER PRIMARY KEY AUTOINCREMENT` | `SERIAL PRIMARY KEY` | All child tables and legacy tables |
| `DATETIME DEFAULT CURRENT_TIMESTAMP` | `TIMESTAMP DEFAULT CURRENT_TIMESTAMP` | All `created_at`/`updated_at` columns |
| `INTEGER NOT NULL` (FK to parent) | `TEXT NOT NULL` | 5 child table FK columns that reference UUID parents |

### BOOLEAN columns

The `hso_equipment_checks.confirmed`, `checkout_items.crew_confirmed`, and `checkout_items.staff_confirmed` columns are declared `BOOLEAN` but the app sends string values. Consider using `TEXT` instead for PostgreSQL compatibility, or adding type conversion in the repository code.

### Query placeholder syntax

All repositories use SQLite `?` placeholders. PostgreSQL requires positional `$1, $2, $3...` placeholders.
